# 風雲戦国見立帖 〜千人一首〜 × プレーリーカード 企画書

## プロジェクト名

**「風雲戦国見立帖 〜千人一首〜」**（ふううんせんごくみたてちょう 〜せんにんいっしゅ〜）

- 略称: **千人一首** / **見立帖**

## 概要

CloudNative Days / SRE Kaigi / Platform Engineering Kaigi 合同イベント（2026年5月14-15日 / 名古屋・中日ホール）にて、
プレーリーカードを活用した心理テスト型エンターテインメント体験を提供する。

参加者がカードを読み取ると、投影法をベースにした心理テストが始まり、
最終的に「戦国人物に見立てたエンジニアタイプ診断」と「AIが詠んだパーソナライズ短歌（千人一首）」、
そして「他の参加者の一首ギャラリー」が表示される。

## イベント背景

- **CloudNative Days / SRE Kaigi / Platform Engineering Kaigi** の3カンファレンス合同開催
- 名古屋で初の合同開催、目標1,000人規模
- 3つの技術領域が交差する場での交流促進が狙い

## デザインコンセプト

**「構造はガチ、結果はネタ」× 和風テイスト**

- イベント全体のデザインコンセプトに合わせ、**和風の世界観**を全体に採用
- 題字・ロゴに毛筆体を使用するイベントのトーンと統一
- 問いの設計には投影法（SCT・P-Fスタディ・リュッシャーカラーテスト）の手法を活用
- 結果は**尾張の三英傑・名古屋ゆかりの歴史上の人物**に紐づける
- カンファレンスの息抜きとして、その場でサクッと楽しめる体験
- 名古屋開催ならではの地域性を活かした演出

## 体験フロー

```
① カード選択（6枚から直感で1枚）
   - 6枚の異なるデザイン / 色のプレーリーカードを並べる
   - 参加者は直感で1枚を選んでNFCで読み取る
   - カード選択自体がリュッシャーカラーテスト的な意味を持つ
        ↓
② プロフィールページ表示（カードごとに異なる内容）
   - 心理テストの「第1問」として機能
   - シナリオ（和風の物語的・抽象的な場面設定）が提示される
   - 4つの選択肢がリンクとして並ぶ
   - P-Fスタディ的な「場面での対処行動」を問う
   - 選択肢はURLパラメータで分岐（実際は同じWebアプリへのリンク）
        ↓
③ Webアプリ：第2問（選択式・分岐型）
   - Q1の回答によって異なる問いが表示される（計24問）
   - シナリオを深掘りする追加の問い（4択）
   - TAT的な要素を含む
        ↓
④ Webアプリ：最終問（自由入力）+ 追加項目
   - SCT（文章完成法）形式
   - シナリオに合わせた未完成文を完成させる
   - 例：「あなたは思わず『　　　』と呟いた。」
   - カードごとに問いのバリエーションが異なる
     - 城の石垣に刻んだ言葉、合戦前夜に書いた書状、茶室で呟いた一言、など
   - 追加項目（任意）：
     - Xアカウント（@xxxxxxx）
     - OGP画像にアイコン・IDを含めるかの選択（チェックボックス）
        ↓
⑤ 結果表示
   - 上段：自分の診断結果
     - 見立て人物名 + 二つ名
     - 自分の自由回答を組み込んだ診断テキスト（事前定義テンプレート）
     - LLMが詠んだパーソナライズ短歌（千人一首）
   - 下段：一首ギャラリー
     - 同じ人物に見立てられた他の参加者の短歌一覧
        ↓
⑥ SNSシェア
   - 結果画面をシェア可能な形式で提供
```

### フォールバック体験フロー（プレーリーカードなしの場合）

プレーリーカードの製作が間に合わない、または予算の都合で作成できない場合の代替手段。
本番では使わない可能性もあるが、Webアプリ側に組み込んでおく。

```
① カード選択ランディングページ（①の代替）
   - 6枚のカードを和風デザインでWeb上に並べる
   - 参加者がタップ / クリックで直感的に1枚を選ぶ
   - 色・デザインの視覚的な選択体験はNFCと同等に保つ
        ↓
② Q1ページ（②の代替）
   - プレーリーカードのプロフィールページ相当
   - 選んだカードに対応するシナリオと4択を表示
   - 選択後、通常フロー③以降に合流
        ↓
③〜⑥ 通常フローと同一
```

**設計方針：**
- 通常フロー（NFC経由）とフォールバック（Web選択経由）は③以降で合流
- URLパラメータで経路を区別可能にしておく（分析用）
- フォールバック経路は `/` でアクセス可能にする

**URL設計：**
- Q1回答後のリンク: `/q2?t=<token>`（4文字英数字の固定トークン）
- 24通り（カード6 × Q1選択肢4）に事前割り当て、推測困難
- プレーリーカードのプロフィールページからも同じURLにリンク
- サーバー側でトークン → (card, q1) をルックアップ
- 無効なトークンはトップページにリダイレクト

## 心理学的根拠

| ステップ | 手法 | 説明 |
|---------|------|------|
| カード選択 | リュッシャーカラーテスト | 色・デザインの直感的選択が心理状態を反映 |
| 第1問 | P-Fスタディ | フラストレーション場面での対処行動から行動傾向を読む |
| 第2問 | TAT（主題統覚検査） | 曖昧な場面に対する解釈から価値観を読む |
| 最終問 | SCT（文章完成法） | 未完成文の補完で無意識の価値観が投影される |

※ あくまでエンタメ利用。学術的な正確性は求めない。

## 分岐の規模感

- カード6種 × 第1問（4択）× 第2問（4択・分岐型）= **96パターン → 32人に割り当て**（1人あたり3パターン）
- 第2問は第1問の回答によって問いが変わる（分岐型）→ 物語の没入感を重視
- 作成する問いの数：Q1が6問 + Q2が24問（6カード×4分岐）= 計30問 + 自由記述テンプレート6種
- 結果テンプレートは32人分を事前に用意
- 最終問の自由回答は結果テンプレートにはめ込み

### LLM短歌生成（千人一首）

結果画面でLLMが**短歌（五七五七七 / 31音）**を一首詠み、参加者にパーソナライズされた一首を贈る。

#### 入力（プロンプトに渡す情報）

| 項目 | 例 |
|------|-----|
| 診断結果の人物名 | 織田信長 |
| 二つ名 | 破壊的リファクタリングの革命児 |
| 人物のテック要素キーワード | リファクタリング、レガシー破壊、本能寺、revert |
| ユーザーの自由記述回答 | 「明日の自分、頼んだぞ」 |
| カードのシナリオテーマ | 合戦前夜の陣中 |

#### 出力

- **短歌1首**（五七五七七 / 31音）
- テック用語と和風の掛詞・本歌取りを組み合わせる
- ユーザーの自由記述の要素を織り込んでパーソナライズ

#### プロンプト設計方針

```
あなたは戦国時代の歌人です。
以下の情報をもとに、テック用語と和歌の技法を融合させた短歌（五七五七七）を一首詠んでください。

【人物】{character_name}（{title}）
【テーマ】{tech_keywords}
【詠み人の言葉】「{user_free_text}」
【場面】{scenario_theme}

制約：
- 五七五七七（31音）を厳守
- テック用語を最低1つ含める
- 詠み人の言葉のエッセンスを反映させる
- 笑いと格調高さを両立させる
- 短歌のみを出力（説明不要）
```

#### ギャラリーの一首

- 他の参加者の結果にもLLMが一首詠んでいる → **同じ人物に見立てられた者たちの一首**としてギャラリー表示
- 表示ルール: 最新2首 + ランダム3首 = 5首（5首未満は全首表示）
- D1に短歌テキストも保存

#### トークン見積もり（更新）

| 項目 | 旧見積もり | 新見積もり |
|------|-----------|-----------|
| input tokens | ~700 | ~800（短歌プロンプト追加分） |
| output tokens | ~250 | ~100（短歌は31音＋句読点のみ、こじつけポエムは事前定義テンプレートに変更） |
| 合計 | ~950 | ~900 |

※ こじつけポエム（診断テキスト）はLLM生成から**事前定義テンプレート + 自由回答はめ込み**に変更。LLMの役割を短歌生成に集中させることで、トークン消費を抑えつつ体験の質を上げる。

### LLM呼び出し戦略

Cloudflare Workersはアジアのトラフィックを香港（HKG）経由でルーティングすることがあり、
OpenAIは香港からのアクセスを403で拒否する（CND2で実際に遭遇した問題）。

**対策：**
1. **Placement Hints**（2026年1月追加の新機能）でWorkerを米国リージョンで実行
2. それでも403が発生した場合、**OpenRouter経由**にフォールバック
3. `request.cf.colo` でデータセンターを検出し、HKG等の場合は即座にフォールバック

```toml
# wrangler.toml
[placement]
region = "aws:us-east-1"
```

### コスト方針・試算

**方針：** CloudFlareを選定した主な理由はコスト最適化。運用経験ベースではない。
より最適な仕組みがあれば柔軟に切り替える方針だが、本プロジェクトの規模ではCloudflareの無料枠で十分。

**前提：** 来場者1,000人 × 参加率50% = 500人

#### ユーザーあたりのリソース消費

| リソース | 1人あたり |
|---------|----------|
| ページ読み込み（静的） | 1回（SPA、以降クライアント遷移） |
| Workers API呼び出し | 3〜4回 |
| D1 書き込み | 2行（Q2回答、自由回答） |
| D1 読み取り | 2クエリ（結果テンプレ、ギャラリー ~20行） |
| LLM呼び出し | 1回（~700 input + ~250 output tokens） |

#### サービス別コスト

| サービス | 無料枠 | 500人の消費量 | コスト |
|---------|--------|-------------|-------|
| Cloudflare Pages | 帯域無制限 | 静的アセット配信 ~500回 | $0 |
| Cloudflare Workers | 100Kリクエスト/日 | ~2,000リクエスト/2日間 | $0 |
| Cloudflare D1 | 5M行読み取り/日, 100K行書き込み/日 | 読み ~10,000行, 書き ~1,000行 | $0 |
| OpenAI gpt-5-mini | Tier1: 2.5M tokens/日 | ~475,000 tokens（無料枠の約10%） | $0 |

|---------|-----------|

#### 保守的試算

楽観的な前提を排除し、以下の条件で再試算：

| 要素 | 標準想定 | 保守的想定 |
|------|---------|-----------|
| 参加率 | 50%（500人） | 80%（800人） |
| 体験回数 | 1回/人 | 2回/人（友達のカードも試す等） |
| LLMトークン | 950/回 | 1,500/回（プロンプト肥大化・出力長め） |
| SNSシェア率 | — | 30%（240人、1シェアあたりOGP取得5リクエスト） |
| 開発テスト | — | 本番前に200回のテスト実行 |
| Bot/クローラー | — | 本番アクセスの20% |

**実効リクエスト数：**
- 体験: 1,600回 × 4 API = 6,400
- SNSクローラー: 240 × 5 = 1,200
- Bot: (6,400 + 1,200) × 20% = 1,520
- 開発テスト: 200 × 4 = 800
- **合計: ~9,920 Workerリクエスト / 2日間（~5,000/日、無料枠の5%）**

**LLMトークン（保守的）：**
- 体験: 1,600回 × 1,500 = 2,400,000 tokens
- テスト: 200回 × 1,500 = 300,000 tokens
- 合計: 2,700,000 tokens → 1日あたり1,350,000（無料枠2.5Mの54%）

| シナリオ | 合計コスト |
|---------|-----------|
| 保守的（無料枠内） | **$0** |
| 保守的（LLM有料時） | **~$0.81（約120円）** |
| 最悪（全OpenRouter + 参加率100% + 3回体験） | **~$2.50（約380円）** |

**結論：どのシナリオでもワンコイン以下。コストリスクはほぼゼロ。**
